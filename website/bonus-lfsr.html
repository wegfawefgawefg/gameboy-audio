How to code GameBoy music.

Bonus: Real Game Boy noise: LFSR.

The random float noise from the noise chapter is already enough to make good effects. This chapter is the more Game Boyish version. The real noise channel does not ask for random.uniform(-1, 1). It uses a tiny bit machine.

The bit machine is called an LFSR. That stands for linear feedback shift register. Gross name. It just means we keep one integer around, pull some bits out of it, combine them, shift the integer over, and feed the new bit back in. That gives us a long stream of high/low offsets that feels noisy.

Before we do the compact bit math version, here is the same idea as a literal list of True and False values. It is easier to read because you can see the slots move. 

register = [
    True, True, True, True,
    True, True, True, True,
]

tap = 3

# Read two slots.
right_bit = register[-1]
tap_bit = register[-1 - tap]

# XOR means "different".
# True  xor True  -> False
# True  xor False -> True
# False xor True  -> True
# False xor False -> False
new_bit = right_bit != tap_bit

# Pop the right side off.
register.pop()

# Push the new feedback bit onto the left side.
register.insert(0, new_bit)

# The new rightmost bit is the high/low speaker offset.
if register[-1]:
    sample = -1
else:
    sample = 1

One tick looks like this.

# before
[True, True, True, True, True, True, True, True]

# tap 3 compares these two:
#                          right_bit
#              tap_bit     |
#              |           |
[True, True, True, True, True, True, True, True]

new_bit = True != True  # False

# shift right, insert False on the left
[False, True, True, True, True, True, True, True]

Python lists are readable, but computers can pack the same list of booleans into one integer. Then shifting the list becomes shifting bits. 

# list version
right_bit = register[-1]
tap_bit = register[-1 - tap]
new_bit = right_bit != tap_bit
register.pop()
register.insert(0, new_bit)

# bitwise version
right_bit = lfsr & 1
tap_bit = (lfsr >> tap) & 1
new_bit = right_bit ^ tap_bit
lfsr = (lfsr >> 1) | (new_bit << 14)

The useful knobs are hold, feedback tap, and the volume shape. Hold changes how many samples reuse one noise bit. Feedback tap changes the pattern of bits. Volume shape decides whether it is a constant hiss or a little hit that dies away. 

sample_rate = 44100
seconds = 0.8
volume = 0.30
num_samples = int(seconds * sample_rate)

samples = []

current_offset = 1
hold_samples = 9
feedback_tap = 1
lfsr = 0b111_1111_1111_1111

for sample_index in range(num_samples):
    # Audio samples still happen 44,100 times per second,
    # but the noise bit only changes every 9 samples.
    # Smaller hold = brighter fizz.
    # Bigger hold = rougher chunk.
    if sample_index % hold_samples == 0:
        # Take two bits from the register.
        # XOR them together.
        # That result becomes the new incoming bit.
        bit = (lfsr ^ (lfsr >> feedback_tap)) & 1

        # Shift everything one slot to the right,
        # then put the new bit back on the left.
        lfsr = (lfsr >> 1) | (bit << 14)

        # The bottom bit becomes our speaker offset.
        current_offset = -1 if lfsr & 1 else 1

    # Constant noise is just hiss.
    # Fade it down and now it starts acting like a tiny drum hit.
    time = sample_index / sample_rate
    fade = max(0, 1 - time / 0.45)

    sample = current_offset * volume * fade
    samples.append(sample)

Heres the bit machine actually doing the thing. The line picks two bit slots, XORs them into a new bit, the register shifts right, and the bottom bit becomes the next high/low sample.

GAY BOY

Thats basically the point of the Game Boy noise channel. Channel 4 does not play normal melody notes like the pulse channels. It advances this little LFSR thing and turns the bit stream into high/low offsets. The hold-ish settings are the character knobs. Not magic. Just tunable trash.

The real Game Boy exposes those knobs through its noise registers. You do not send it a huge list of random samples. You set parameters like timer rate, width mode, and envelope, and the chip spits out the noisy high/low stream for you.

But in software we are not stuck with the Game Boy circuit. XOR is special because it tends to make long useful bit patterns. But you can put other little logic functions in the feedback path and listen to the machine get weird, collapse, buzz, or turn into a different texture. 

def feedback(mode, a, b, c):
    if mode == "xor":
        return a ^ b
    if mode == "nand":
        return 1 - (a & b)
    if mode == "xor_or":
        return (a ^ b) | c
    if mode == "majority":
        return 1 if a + b + c >= 2 else 0
    if mode == "parity3":
        return a ^ b ^ c


new_bit = feedback(mode, bit_0, bit_tap_a, bit_tap_b)
register = (register >> 1) | (new_bit << (length - 1))

Then theres the other chain of thought. What if the register is not bits. What if it is a row of smooth numbers, and the feedback function is something soft like tanh, sin, or a folded curve. Now it is not a Game Boy noise channel anymore. Its a little procedural noise synth. 

def soft_xor(a, b, c, drive):
    return tanh((a - b) * drive)


def sine_mix(a, b, c, drive):
    return sin(a * drive + b * drive * 0.7 + c * drive * 0.35)


new_value = feedback(a, b, c, drive)
state = [new_value] + state[:-1]
