How to code GameBoy music.

Bonus: Sampling real sounds, and the wavetable

Why Do We Even Want This?

We have two pulse voices and noise now. Thats enough to make something gameboy-ish, but it is not the full machine.

The Game Boy has four normal sound channels. Pulse 1. Pulse 2. Wave. Noise. So if the song wants a third pitched part, it cannot ask for another pulse voice. There are only two. The third pitched part has to use the weird one: the wave channel.

Route 3 is a nice example because you can hear the obvious square-ish flutes, you can hear the fuzzy snare, and there is still another pitched line in the stack. That extra line is channel 3. The annoying thing is that channel 3 is not a pulse channel. Its a little 32-sample table.

This Is Not Raw Hardware Code

Important correction before we pretend we understand anything. This Route 3 file is not the CPU manually writing every Game Boy sound register by hand. The Pokemon devs already built a little music language.

The song says friendly stuff like note E_, 6, vibrato 4, 1, 0, note_type 12, 1, 2, and octave 4. Then the audio engine reads those commands and turns them into actual hardware state. So this assembly is already one abstraction above the chip. 

song data
    note_type 12, 1, 2
    octave 4
    note E_, 1

music driver
    remembers note speed
    remembers current octave
    applies vibrato
    chooses CH3 wave instrument
    calculates hardware frequency

Game Boy APU registers
    CH3 enable
    CH3 volume level
    CH3 frequency
    CH3 wave RAM

octave 4

Sets the current octave for following notes. After this, note E_ means E in octave 4 until another octave command changes it.

vibrato 4, 1, 0

Tells the driver to wiggle the pitch. Delay first, then extent/rate stuff. You write one command; the driver keeps nudging frequency while notes play.

note_type 12, 1, 2

Sets note timing and channel settings. On pulse channels this mostly feels like volume/envelope. On CH3, that last value is special: it picks the wave instrument.

That last one matters for this chapter. In the Red/Blue audio engine, when the current channel is CH3, the fade slot is reused as a wave instrument number. The macro packs volume and fade into one byte, and the engine stores that low nibble as wMusicWaveInstrument. Route 3 uses note_type 12, 1, 2, so the wave channel is using wave instrument 2. Macro source. Engine source. 

; audio/engine_1.asm, simplified
; if channel 3, store the packed byte's low nibble as wave instrument
ld hl, wMusicWaveInstrument
ld a, d
and $f
ld [hl], a

And that instrument number points at one of the predefined 32-nibble wave shapes in audio/wave_samples.asm. Wave source.

Red/Blue does not have a giant library here. It has wave0 through wave4, then wave5 for the spookier Lavender/Pokemon Tower style stuff, and the remaining pointer slots just point back at wave5. Each normal wave costs only 16 bytes, but this engine is still very much "pick from a tiny instrument shelf", not "invent a new waveform every note." 

; wave instrument 2
; 32 tiny values, each 0..15
.wave2
    dn  1,  3,  6,  9, 11, 13, 14, 14
    dn 14, 14, 15, 15, 15, 15, 14, 13
    dn 13, 14, 15, 15, 15, 15, 14, 14
    dn 14, 14, 13, 11,  9,  6,  3,  1

So yeah, CH3 is more different than just "another note list." The notes choose pitch and timing. The note_type command quietly chooses which 32-value waveform that pitch is played through.

Also, this is why you do not see procedural noise in this file. Noise is not a wavetable instrument in Pokemon Red/Blue. Noise is CH4. The song says drum_note 17 or drum_note 19, and the driver turns that into noise-channel settings. The random-ish part comes from the Game Boy noise hardware, not from a list of random wave RAM values.

So What Is A Wavetable?

A wavetable is just a tiny waveform you store in a list. Instead of saying "calculate a square wave with a formula", you say "here are 32 offsets, keep reading them in a loop." 

# One cycle of a tiny wave.
# These are still speaker offsets.
wave_table = [
    -1.00, -0.75, -0.50, -0.25,
     0.00,  0.25,  0.50,  0.75,
     1.00,  0.75,  0.50,  0.25,
     0.00, -0.25, -0.50, -0.75,
]

# sample_index tells us which output sample we are filling.
# frequency tells us how many times per second we should loop the table.
def wavetable(sample_index, frequency, volume):
    table_position = sample_index * frequency * len(wave_table) / sample_rate

    table_index = int(table_position) % len(wave_table)
    offset = wave_table[table_index]

    return offset * volume

The Index Is The Whole Trick

The line to stare at is the table position line. It answers one boring question: after this many output samples, how far through the tiny table should we be? 

table_position = sample_index * frequency * len(wave_table) / sample_rate
table_index = int(table_position) % len(wave_table)

# Example with 16 table values:
#
# table_position 0.0  -> table index 0
# table_position 4.2  -> table index 4
# table_position 8.9  -> table index 8
# table_position 15.4 -> table index 15
# table_position 16.0 -> table index 0 again
# table_position 17.0 -> table index 1 again

The % len(wave_table) part is the loop. Fall off the end of the table, wrap back to the front. Thats it.

Now Make It More Game Boy-ish

The actual Game Boy wave channel stores 32 tiny values. Each value is only four bits, so it can only be a number from 0 to 15. That is crude as hell, but it is enough to draw a shape. And the shape does not have to be square. Thats the point. 

sample_rate = 44100

# 32 tiny 4-bit values.
# 15 means high offset. 0 means low offset.
wave_ram = [
    15, 15, 15, 15, 15, 15, 15, 15,
     0,  0,  0,  0,  0,  0,  0,  0,
    15, 15, 15, 15, 15, 15, 15, 15,
     0,  0,  0,  0,  0,  0,  0,  0,
]

def wave_channel(sample_index, frequency, volume):
    table_position = sample_index * frequency * 32 / sample_rate
    table_index = int(table_position) % 32

    four_bit_value = wave_ram[table_index]

    # Convert 0..15 into roughly -1.0..1.0.
    offset = (four_bit_value / 15) * 2 - 1

    return offset * volume

In Route 3 it is being used like a cheap third pulse-ish voice. That can feel boring, but it is still useful because the Game Boy only has two real pulse channels. The wave channel lets the song sneak in one more pitched part.

The more general reason you want this channel is that the table is yours. You can draw a bass shape, a nasal lead, a fake pulse, a tiny organ-ish thing, or some ugly little custom waveform. CH1 and CH2 give you four fixed pulse shapes. CH3 says fine, draw your own 32-value trash rectangle.

Can We Race The Table?

Yeah. Sort of. The normal use is "load one table, loop it as a tone." But you can also keep replacing the table while the song runs. Then the wave channel starts acting less like a synth oscillator and more like terrible 4-bit sample playback.

So lets use actual audio samples. Not fake math. First lets do the thing we actually care about: listen to the original Pikachu sound from Yellow, then take our clean pikachu.wav and crush it until it starts living in that same neighborhood.

This is also the trick behind those tiny voice clips you hear in some Game Boy games. The voice is not magic. Its a tiny voice sample shoved through a very small, very crunchy audio path.

We tried the fancy wavetable version first and it sounded wrong. Too good. Thats weird, so we looked at the Pokemon Yellow source. Turns out they did not race the wavetable for Pikachu. They did something much nastier.

Yellow stores the Pikachu source clips as 22050 Hz, 8-bit, mono WAVs in the disassembly, but the build tool packs them down to 1-bit PCM. Each sample is only above center or below center. Eight samples fit in one byte. Thats a huge space win when you have 42 Pikachu clips. Converter source. Playback source.

So yeah, they made it worse than it had to be. Not because they were stupid. Because ROM space is real, and a recognizable crunchy Pikachu is better than no voiced Pikachu. 

# Pokemon Yellow style, simplified from tools/pcm.c
def convert_to_yellow_1bit(source_samples):
    bits = []

    for sample in source_samples:
        if sample > 0:
            bits.append(1)
        else:
            bits.append(0)

    return bits


# Simplified from PlayPikachuPCM.
# The real code uses CH3, fills wave RAM with 0xff,
# then toggles rAUD3LEVEL between mute and loud.
def play_yellow_1bit(bits):
    output = []

    for bit in bits:
        if bit == 1:
            output.append(1.0)
        else:
            output.append(-1.0)

    return output
target = load_wav("original_pikachu_sound_from_yellow.wav")

Our Source

Thats the focused lesson: for Pikachu, the nasty sound is mostly the 1-bit conversion and the volume-toggle playback routine. Now here is the separate wavetable-racing idea we were playing with earlier. This is still a useful Game Boy trick to understand, but it is not what Yellow uses for Pikachu. 

def four_bit(value):
    # value starts as -1.0..1.0
    # store it as 0..15 because thats all wave RAM can hold
    return round((value + 1) * 7.5)


def stream_samples_through_wave_ram(source_samples):
    output = []
    wave_ram = [8] * 32

    for sample_index, source_sample in enumerate(source_samples):
        # Every 32 output samples, race in a new wave table.
        if sample_index % 32 == 0:
            chunk = source_samples[sample_index : sample_index + 32]
            wave_ram = []

            for value in chunk:
                wave_ram.append(four_bit(value))

        four_bit_value = wave_ram[sample_index % 32]
        offset = (four_bit_value / 15) * 2 - 1

        output.append(offset)

    return output

The fight voice, laser, glass, and footstep are from Kenney's CC0 audio packs. The scream and cat are from the CC0 Sound Effects 2 pack. The roar is from SSE Library: ANIMALS, and the crash is from SSE Library: VEHICLES. We trimmed everything short, converted it to mono, gated/denoised the quiet junk, then ran the Wave RAM abuse pass above.

So the wavetable is not just "a third square wave, but worse." It is a tiny programmable shape buffer. You can use it politely as a looping waveform, or abuse it as garbage sample playback.

Rendering A Note With A Wavetable

This looks almost exactly like the sequence chapter. The only difference is that the voice function reads from a table instead of calculating a square wave. 

sample_rate = 44100

def wave_channel(sample_index, frequency, volume, wave_ram):
    table_position = sample_index * frequency * len(wave_ram) / sample_rate
    table_index = int(table_position) % len(wave_ram)

    four_bit_value = wave_ram[table_index]
    offset = (four_bit_value / 15) * 2 - 1

    return offset * volume


def render_note(output, note_name, start_time, duration, volume, wave_ram):
    frequency = note_frequency[note_name]
    start_index = int(start_time * sample_rate)
    num_samples = int(duration * sample_rate)

    for sample_index in range(num_samples):
        output_index = start_index + sample_index

        if output_index >= len(output):
            break

        output[output_index] += wave_channel(
            sample_index,
            frequency,
            volume,
            wave_ram,
        )

Same old sample list. Same old +=. The voice just gets its shape from a list now.
